# Deployment Guide

This guide explains how to build, publish, and distribute the CMatrix application using Docker containers.

## Prerequisites

1.  **Docker**: Ensure Docker and Docker Compose are installed.
2.  **Docker Hub Account**: You need an account on [Docker Hub](https://hub.docker.com/) (or another container registry).
3.  **Login**: Make sure you are logged in to Docker Hub in your terminal:
    ```bash
    docker login
    ```

## Publishing the Application

We have provided a script to automate the build and push process.

1.  **Run the publication script**:
    ```bash
    ./publish_images.sh
    ```
    *   You will be prompted to enter your **Docker Hub username**.
    *   You can optionally specify a **version tag** (e.g., `v1.0.0`). If you leave it blank, it defaults to `latest`.

2.  **What the script does**:
    *   Builds the `backend` and `frontend` Docker images.
    *   Pushes them to your Docker Hub repository (e.g., `username/cmatrix-backend:latest`).
    *   Generates a `docker-compose.release.yml` file configured to use these public images.

## Distributing to Users

To let others use your application, they **do not** need the source code. They only need:

1.  The `docker-compose.release.yml` file (generated by the script).
2.  A `.env` file (optional, but recommended for secrets).

### Instructions for End Users

Share the following instructions with people who want to run your app:

1.  **Download** the `docker-compose.release.yml` file.
2.  **Create a folder** (e.g., `cmatrix`) and place the file inside.
3.  **(Optional) Create a `.env` file** in the same folder if you need to override default settings (like API keys):
    ```env
    HUGGINGFACE_API_KEY=your_key_here
    SECRET_KEY=your_secure_secret
    ```
4.  **Run the application**:
    ```bash
    docker-compose -f docker-compose.release.yml up -d
    ```
5.  Access the app at `http://localhost:3000`.

## Updating the Application

To ship an update:
1.  Make your code changes.
2.  Run `./publish_images.sh` again (you can use a new version tag).
3.  Share the updated `docker-compose.release.yml` (or tell users to pull the new images if using `latest`).
